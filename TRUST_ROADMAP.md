## Roadmap

### Model Diversity (Priority: High)

**Problem:** All reviewers share the same base model (Opus), so "independent"
reviews have correlated biases. 5 Opus reviewers catching the same bug is
less valuable than 4 Opus + 1 different-architecture model catching different bugs.

**Proposed solution:** Assign 1-2 reviewer slots to a fundamentally different
model architecture (e.g., OpenAI Codex, Google Gemini, or an open-source model
via API). The key is architectural diversity — a smaller Claude model (Sonnet)
still shares the same training pipeline and biases. A completely different model
(different training data, different architecture) maximizes the probability of
catching errors the Opus reviewers systematically miss.

**Implementation:**
- Add a `model_diversity` option to Phase 1 persona selection
- When enabled, assign the Devil's Advocate and one specialist to the
  alternative model
- Track which model caught which findings in the report — this builds
  empirical data on whether diversity actually helps

**Trade-off:** Different models may produce lower-quality individual reviews.
But diversity of errors is more valuable than uniformity of quality.

**Effort:** Medium — requires API access to alternative model, plus prompt
adaptation (different models may need different prompt styles).

### Synthetic Benchmark Suite (Priority: High)

**Problem:** Panel scores (e.g., "7/10") are uncalibrated — there's no ground
truth to know if higher scores correlate with fewer actual bugs.

**Proposed solution:** Build a test corpus of 20-50 code samples with known bugs:

```
benchmark/
├── easy/          # Obvious bugs (null deref, SQL injection)
├── medium/        # Subtle bugs (off-by-one, race conditions)
├── hard/          # Deep bugs (temporal logic, distributed systems)
└── clean/         # Bug-free code (tests false positive rate)
```

Run the panel on each sample. Measure:
- **Detection rate** by difficulty tier
- **False positive rate** on clean code
- **Score calibration** — does higher score predict fewer bugs?
- **Finding accuracy** — what % of reported findings are real?
- **Model diversity impact** — does the alternative model catch different bugs?

**Deliverable:** Calibration footnote in every report:
```
Panel Calibration (based on N=50 benchmark):
- Detection rate: 95% easy / 60% medium / 25% hard
- False positive rate: 12%
- Score accuracy: 7+ correlates with <2 real bugs (72%)
- Last calibrated: 2026-03-20
```

**Effort:** 2-3 days for initial corpus + automation script.
Re-calibrate quarterly or after major skill updates.

### Calibration Footnote in Reports (Priority: Medium)

**Depends on:** Synthetic Benchmark Suite (above).

Once benchmark data exists, add a `## Panel Calibration` section to the
Phase 6 report template. This transforms the panel from "trust it because
it sounds authoritative" to "trust it because we measured its accuracy."

**Effort:** Low (once benchmarks exist) — template change only.

## Explicitly Deferred

These were evaluated and deemed low-ROI for the panel's use case:

| Item | Why Deferred |
|------|-------------|
| Provenance graph | Users care about the verdict, not the deliberation process. Transcript is already available. |
| Run-over-run stability | Expensive (run panel twice) and stability ≠ accuracy. Better to invest in benchmarks. |
| Confidence decay | Reviews are point-in-time; they don't "age" meaningfully. |
| User correction feedback loop | Panel runs ad-hoc, not continuously. No natural feedback path without overengineering. |
