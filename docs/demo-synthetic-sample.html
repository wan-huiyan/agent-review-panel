A1 P0 VR_CONFIRMED client_secret stored in session cookie without encryption
src/auth/session.py:28-32 — Flask's itsdangerous signs but does not encrypt the session cookie. The OAuth client_secret is readable in browser devtools as plaintext base64 JSON.
Raised by: SA CH DA 3 reviewers · DA upgraded P1→P0 after debate
📖 Narrative

The ExampleApp Payment Service uses Flask's default signed session cookie to persist OAuth state across redirects. At src/auth/session.py:28, the code writes the full token payload into session['oauth_token'] — including client_secret, which it pulls from os.getenv('OAUTH_CLIENT_SECRET'). Flask's itsdangerous layer signs the cookie for integrity but does NOT encrypt it. The payload is base64-encoded JSON that any client can decode without knowing the signing key.

This is a classic Flask default that catches security-inexperienced developers repeatedly: if you need confidentiality for session contents, you need server-side session storage (Flask-Session with Redis) or an encrypted cookie implementation. The signing layer protects against cookie modification, not reading. Any user running the app in Chrome can open DevTools → Application → Cookies and read client_secret in plain text within two clicks.

The blast radius is the full OAuth 2.0 client credential, which alongside the client_id allows an attacker to impersonate the application against the OAuth provider's token endpoint. For a payment service backed by third-party processor credentials, this is equivalent to exposing the merchant API key.

B1 P1 VR_CONFIRMED data mask + reindex silently reintroduces filtered rows as zero-revenue days
src/analytics/pipeline.py:915-918 — apply_date_mask removes rows; downstream prep_df calls reindex(full_date_range).fillna(0) which reintroduces them as zero values. Anti-conservative bias (inflates estimated lift).
Raised by: CH AC DA 3 reviewers · 3-round debate on bias direction
📖 Narrative

This is a composition bug: apply_date_mask and prep_df are each individually correct, but the two functions fail together when composed. apply_date_mask removes rows for dates outside the analysis window (returning a sparse DataFrame). prep_df then calls reindex(full_date_range) to align the index to a complete date range, and fillna(0) to clean up missing values for downstream arithmetic. The intent of each step is fine in isolation.

The problem: the filtered rows removed by apply_date_mask become NaNs after reindex, and fillna(0) converts them to zero-revenue days. The downstream lift estimator then treats these fake zero-days as real observations in the control period, artificially suppressing the control baseline. Since the lift is computed as (treatment - control) / control, a suppressed control baseline inflates the reported lift.

This is an anti-conservative bias — exactly the kind most dangerous in campaign-lift analytics, because it pushes results in the direction that users (and stakeholders) are hoping to see. A conservative bias would make findings harder to publish; an anti-conservative bias makes them easier. The magnitude depends on the ratio of masked days to kept days; for the default mask this is roughly 15% inflation of the reported lift.

📄 Code Evidence

def prep_df(df: pd.DataFrame, full_idx: pd.DatetimeIndex) -> pd.DataFrame:

    df['revenue'] = df['revenue'].fillna(0)

    return df
src/analytics/pipeline.py 842-851 The upstream apply_date_mask function — correct in isolation 
def apply_date_mask(df: pd.DataFrame, mask_start, mask_end) -> pd.DataFrame:
    """Remove rows in the masked interval (e.g. holiday dead-zone)."""
    mask = (df.index >= mask_start) & (df.index <= mask_end)
    return df.loc[~mask]

⚖️ Judge Ruling
SJ Supreme Judge

B1 is confirmed at P1. This is exactly the class of composition bug that Phase 2 Data Flow Trace is designed to catch — both apply_date_mask and prep_df are individually correct. The bug lives in the invariant gap between them: one function removes rows, the other fills missing indexes, and neither function knows about the other's semantics.

The anti-conservative bias direction makes this a P1 rather than a P2. Numerical bugs that push results toward favorable outcomes are more dangerous than those pushing away, because stakeholders have no incentive to question a pleasant-looking number. The magnitude (14.8% on the synthetic test case) is meaningful but not catastrophic — hence P1 not P0.

Severity confirmed at P1. Noted: this is a textbook composition bug — the failure mode only appears when both functions run in sequence.
🛠️ Fix Recommendation

Fix prep_df to respect the mask bounds. The function needs to know which index positions are genuinely missing (should be filled with zero) versus intentionally masked (should be dropped from the output). Pass the mask metadata through the call chain.

Before
def prep_df(df, full_idx):
    df = df.reindex(full_idx)
    df['revenue'] = df['revenue'].fillna(0)
    return df
After
def prep_df(df, full_idx, mask_bounds=None):
    if mask_bounds:
        full_idx = full_idx[
            (full_idx < mask_bounds[0]) |
            (full_idx > mask_bounds[1])]
    df = df.reindex(full_idx)
    df['revenue'] = df['revenue'].fillna(0)
    return df

C1 P1 N+1 query pattern in order history endpoint
src/api/routes.ts:142-156 — /api/orders/history loads orders in a loop, then fetches line items individually per order. For 50+ orders, this produces 51 DB queries per page load (O(N) scaling).
Raised by: AC CH 2 reviewers · 1 round on severity
📖 Narrative

The /api/orders/history endpoint implements pagination by loading a page of orders, then looping over each order and fetching its line items with a separate db.lineItems.findMany call. This is a textbook N+1 query pattern: for a user with 50 orders on their history page, the endpoint issues 51 database queries per request (1 for the orders, 50 for the line items).

The impact is bounded — it doesn't corrupt data or break correctness — but it scales linearly with order history, and the order history page is the most-visited customer-facing route after checkout. At 50 orders the p95 latency is already 1.2 seconds. For power users with 200+ orders (wholesale buyers, resellers), this becomes visibly slow and generates unnecessary DB load that cascades to connection-pool pressure during peak traffic.

📄 Code Evidence
src/api/routes.ts 142-156 The classic N+1 loop — one findMany call per order 
router.get('/api/orders/history', async (req, res) => {

    where: { userId: req.user.id },
    orderBy: { createdAt: 'desc' },

  for (const order of orders) {
    const items = await db.lineItems.findMany({
      where: { orderId: order.id },

  }

  res.json(enriched);
});
