# Archived: 2026-04-27 README restructure (rejected)

This directory preserves a rejected proposal for posterity, so future contributors can see what was considered, what the review panel found wrong with it, and what shipped instead.

## Files

- **`proposal.md`** — The original restructure proposal. Suggested deleting three items (CLI equivalent block, `@<marketplace-name>` callout, "Why the marketplace path?" paragraph) from the Installation section, framed as "concrete duplication audit."
- **`panel-verdict.md`** — Output of the multi-agent review panel (Phase 15.1 report, generated by `roundtable:agent-review-panel` running on this very repo). Final verdict: **Reject. 4/10. High confidence.**

## Why this was rejected

Three reasons surfaced by the panel:

1. **Decision-reversal without process** — the proposal's three deletions exactly invert PR #20's documented "Kept" list (CHANGELOG.md:171, also at HOW_WE_BUILT_THIS.md:726, both written 8 days earlier). The proposal cited neither.
2. **CI lint silent neutering** — `scripts/release-check.sh:80` greps for the literal string `"marketplace name is"` (the exact callout the proposal would delete). Deletion would have made the grep return empty, the lint would still print `pass`, and CI would silently allow the deletion.
3. **Inflated savings** — the "~50 lines saved" claim was loose; realistic net savings ~20 lines.

## What shipped instead

[PR #36](https://github.com/wan-huiyan/agent-review-panel/pull/36) — `docs: tighten README prose + add two-way invariant marker`:

- Tightened prose in place (slash-command callout 5 sentences → 1, etc.) without deleting any of PR #20's preserved items
- Added a two-way invariant: HTML comment marker pair `<!-- release-check:marketplace-name-callout -->` in the README, paired with a marker-presence assertion in `release-check.sh`. Now if the callout is ever deleted, CI fails.

## Process lesson

This panel run was self-referential — the project's own `roundtable:agent-review-panel` skill reviewed a proposal about its own README. The panel's strongest finding (the CI-coupling neutering) came from the Phase 8 Completeness Audit, not from the four panel reviewers. That argues for keeping the audit phase as a non-optional checkpoint when proposals touch files referenced by tooling.
